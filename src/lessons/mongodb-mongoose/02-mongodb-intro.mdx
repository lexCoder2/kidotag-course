**MongoDB** es la base de datos que usa kidotag10. En lugar de tablas con filas y columnas, almacena **documentos** JSON en **colecciones**.

## SQL vs MongoDB

| SQL (relacional)  | MongoDB (documental) |
| ----------------- | -------------------- |
| Base de datos     | Base de datos        |
| Tabla             | Colección            |
| Fila              | Documento            |
| Columna           | Campo                |
| JOIN entre tablas | populate() o embed   |
| Schema rígido     | Schema flexible      |

## Documentos BSON

```json
// Documento Alumno en kidotag10
{
  "_id": "ObjectId('64ab12...7f')", // ID único generado por MongoDB
  "nombre": "Ana García",
  "matricula": "2024-001",
  "grado": "3A",
  "activo": true,
  "tags": ["nfc-registrado", "beca"], // Arrays son nativos
  "tutor": "ObjectId('64cd...')", // Referencia a otro documento
  "contacto": {
    // Subdocumento embebido
    "telefono": "555-1234",
    "email": "papa@email.com"
  },
  "createdAt": "2024-01-15T10:30:00Z"
}
```

## Por qué MongoDB para kidotag10

**Un registro de asistencia en SQL** requeriría 3 tablas + JOINs. En MongoDB, un documento puede guardar todo lo necesario para una consulta de asistencia:

```json
// MongoDB — un solo documento con toda la info
{
  "_id": "...",
  "alumno": {
    "nombre": "Ana García",
    "grado": "3A",
    "uid_nfc": "04:AB:CD:12:34"
  },
  "tipo": "entrada",
  "timestamp": "2024-01-15T08:05:00Z",
  "escuela": "Demo Primaria"
}
```

function find(coleccion, filtro = {}) {
  return coleccion.filter((doc) => Object.entries(filtro).every(([k, v]) => doc[k] === v));
}

function findOne(coleccion, filtro) {
  return coleccion.find((doc) => Object.entries(filtro).every(([k, v]) => doc[k] === v));
}
