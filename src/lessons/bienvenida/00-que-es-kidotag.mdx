# ¿Qué es KidoTag?

**KidoTag** es una aplicación web completa para la gestión escolar. Permite a escuelas registrar alumnos, controlar asistencias mediante lectores NFC/RFID, y comunicar información a tutores en tiempo real.

<Callout variante="kidotag" titulo="El proyecto que aprenderemos">
  Todo este curso está basado en el código real de **kidotag10**, un sistema que
  incluye: una API REST, una base de datos MongoDB, autenticación segura y una
  interfaz web en React. Al terminar el curso, entenderás **cada línea** de ese
  proyecto.
</Callout>

## Arquitectura general

El sistema tiene tres grandes partes:

  titulo="Componentes principales de KidoTag"
  diagrama={`

  A[🌐 App Web React] -->|HTTP / WebSocket| B[🖥️ API Express]
  C[📟 ESP32 NFC] -->|HTTP| B

  B -->|JWT| A
  B -->|Socket.IO| A

| Capa          | Tecnología         | Propósito                   |
| ------------- | ------------------ | --------------------------- |
| Frontend      | React + Vite       | Interfaz de usuario         |
| Backend       | Express + Node.js  | API REST                    |
| Base de datos | MongoDB + Mongoose | Almacenamiento              |
| Hardware      | ESP32 + PN532      | Lector NFC                  |
| Tiempo real   | Socket.IO          | Notificaciones instantáneas |

## ¿Qué aprenderás?

Este curso está dividido en **11 módulos** con más de 50 lecciones interactivas:

1. **Backend con Express** — Crear rutas, middlewares y controladores
2. **MongoDB con Mongoose** — Modelos, esquemas y consultas
3. **Autenticación JWT** — Seguridad y sesiones
4. **Documentación con Swagger** — APIs profesionales
5. **Tiempo real con Socket.IO** — WebSockets desde cero
6. **React fundamentos** — Componentes, props y estado
7. **React avanzado** — Hooks, contexto y patrones
8. **Integración web-API** — Conectar frontend y backend
9. **Estilos y UX** — CSS variables y diseño responsivo
10. **Proyecto final** — Reproducir KidoTag completo

## Flujo completo: registro de asistencia

Para entender por qué existen todas estas capas, observa cómo fluye un evento de
asistencia desde el hardware hasta el tutor:

<Mermaid
  titulo="Flujo: el alumno pasa su tarjeta NFC"
  diagrama={`
sequenceDiagram
  participant ESP as 📟 ESP32 (NFC)
  participant API as 🖥️ API Express
  participant DB  as 🗄️ MongoDB
  participant WS  as ⚡ Socket.IO
  participant APP as 🌐 App React

ESP->>API: POST /asistencias { uid_nfc: "A1B2C3" }
API->>DB: Buscar alumno con uid_nfc = "A1B2C3"
DB-->>API: { nombre: "Ana García", grupo: "3°A" }
API->>DB: Guardar { alumno, hora, tipo: "entrada" }
DB-->>API: Asistencia guardada ✅
API->>WS: emit("asistencia", { alumno, hora })
WS->>APP: Notificación en tiempo real
APP->>APP: Actualizar tabla de asistencias
`}
/>

<Callout variante="tip" titulo="¿Por qué necesitamos todas estas piezas?">
  Cada capa tiene una responsabilidad específica: el **ESP32** solo lee
  tarjetas, la **API Express** aplica las reglas del negocio, **MongoDB** guarda
  el historial, y **Socket.IO** evita que el maestro tenga que recargar la
  página. Este patrón de separación se llama *arquitectura en capas*.
</Callout>
